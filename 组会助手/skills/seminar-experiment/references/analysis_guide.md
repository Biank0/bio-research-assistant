# 分析操作详细指南

本文件在 action=analyze 时按需加载，提供完整的分析约束和方法论。

---

## 分析流程

### Step 1: 确认分析范围
- 确认分析模式（failure_pattern / condition_compare / technique_summary / progress_assessment）
- 确认涉及的课题线和时间范围
- 如用户未指定，从上下文推断；推断不出则简短确认一次

### Step 2: 数据收集
- 从 `{experiment_dir}/logs/` 读取相关实验记录
- 从 `{experiment_dir}/threads/` 读取时间轴上下文
- 从 `{experiment_dir}/rules.md` 读取已有经验规则
- 统计纳入分析的实验数量，明确告知用户样本量

### Step 3: 模式识别与分析

#### failure_pattern 模式
参考 `references/failure_analysis_template.md`，执行：
1. 列出所有失败实验的关键条件
2. 找共性特征（共同条件、共同现象、时间模式）
3. 与成功实验对比，识别关键差异变量
4. 生成假设排序表（假设 + 支撑 + 反驳 + 验证方法）
5. 给出分级建议（立即可做 / 需要准备 / 需要讨论）

#### condition_compare 模式
1. 构建条件-结果矩阵表
2. 标注每个条件变量的变化范围
3. 识别结果转折点（从失败到成功的条件阈值）
4. 输出最优条件组合建议（标注置信度）

#### technique_summary 模式
1. 聚合某技术的所有使用记录
2. 总结成功条件范围和常见失败原因
3. 提炼为 protocol 建议（最佳实践 + 避坑点）
4. 与文献中推荐的条件对比（如有 seminar-literature 记录）

#### progress_assessment 模式
1. 梳理课题各线索的推进状态
2. 计算各线索的"有效实验比"（成功+有信息量的失败 / 总次数）
3. 识别瓶颈线索和推进最快的线索
4. 建议资源重新分配（时间、精力侧重）

### Step 4: 经验规则提炼
- 从分析中提炼新规则 → 追加到 `{experiment_dir}/rules.md`
- 规则格式参考 `references/experience_rules_template.md`
- 检查是否与已有规则冲突——冲突时更新旧规则状态

### Step 5: 产出与确认
- 展示分析报告全文
- 用户确认后写入 `{experiment_dir}/analysis/{date}_{type}.md`
- 如提炼出新规则，追加到 `rules.md`（规则数从该文件现读现算，不入 memory）

## 分析约束（强制）

### 证据与推理
- MUST 每个结论标注支撑证据：`[来源: {date}_{experiment}]`
- MUST NOT 把相关性断言为因果——使用"可能相关""需要对照验证"
- MUST 数据不足时降级结论：标注"基于 N 次实验，置信度: 低"
- MUST NOT 用领域通用知识替代实际实验数据做推断
- MUST 区分"实验数据显示"和"一般认为"

### 表述规范
- 条件句 > 绝对句："当X条件下Y更可能" > "X导致Y"
- 不确定就说不确定，永远比错误的确定性好
- 定量优于定性："3次中2次成功" > "成功率较高"

### 分析局限声明
每份分析报告末尾 MUST 包含：
- 样本量
- 未控制变量
- 结论适用范围
- 哪些额外实验能提升置信度

## 经验规则管理

### 规则生命周期
```
观察 → 假设（低置信度）→ 初步验证（中）→ 多次验证（高）→ 反例出现（修正）→ 适用条件缩窄
```

### 规则冲突处理
- 新数据与旧规则矛盾时，不直接删除旧规则
- 标记旧规则为"待修正"，记录矛盾实验
- 尝试找到更精确的适用条件来调和
- 无法调和时，保留两条并标注"互斥条件"

### 规则回顾
- 每次 review 时，顺带检查已有规则是否仍然成立
- 3 个月未被验证/引用的规则，标记为"休眠"
